## 一、主流 Agent 框架对比

### 1.1 LangChain

**定位**：LLM 应用开发框架，提供组件化构建块

**核心能力**：
- **Chains**：多步骤任务编排（顺序/并行）
- **Agents**：基于 LLM 决策的工具调用
- **Memory**：对话历史/向量存储
- **Tools**：预置工具集（搜索、代码执行、API 调用）

**任务规划模式**：
```python
# ReAct 模式：Reason + Act
agent = AgentExecutor(
    agent=ReActAgent.from_llm(llm, tools),
    verbose=True
)
# LLM 自主决定：思考→行动→观察→循环
```

**适用场景**：
- ✅ 需要精细控制执行流程
- ✅ 复杂多步骤任务编排
- ✅ 需要自定义工具集成
- ❌ 学习曲线陡峭，代码量大

---

### 1.2 Dify

**定位**：可视化 AI 应用开发平台

**核心能力**：
- **Workflow**：拖拽式工作流编排
- **Agent Mode**：自主工具调用
- **RAG Pipeline**：知识库检索增强
- **API 发布**：一键部署为 API

**任务规划模式**：
```yaml
# 可视化编排
workflow:
  - node: LLM
    prompt: "分析用户需求"
  - node: Condition
    branches:
      - if: "需要搜索" → Tool: WebSearch
      - else: → LLM: 直接回答
  - node: Output
```

**适用场景**：
- ✅ 快速原型开发
- ✅ 非技术人员参与
- ✅ 需要可视化调试
- ❌ 自定义逻辑受限

---

### 1.3 OpenClaw Subagent

**定位**：本地优先的多 Agent 协作系统

**核心能力**：
- **Subagent  spawn**：主 Agent 动态创建子 Agent
- **Push-based 结果**：子任务完成自动汇报
- **并行调度**：多 subagent 同时执行
- **本地工具链**：文件、浏览器、终端、Feishu

**任务规划模式**：
```markdown
# 主 Agent 决策
分析任务 → 拆解子任务 → spawn subagent → 等待完成 → 汇总

# Subagent 执行
接收任务 → 自主执行 → 完成汇报（不轮询）
```

**架构特点**：
```
┌─────────────────────────────────────────┐
│           Main Agent                    │
│  (决策中枢 / 任务拆解 / 结果汇总)         │
└──────────────┬──────────────────────────┘
               │ spawn
               ▼
┌─────────────────────────────────────────┐
│  Subagent 1    Subagent 2    Subagent 3 │
│  (写脚本)      (生成图)      (剪视频)    │
│  并行执行       并行执行       并行执行   │
└─────────────────────────────────────────┘
```

**适用场景**：
- ✅ 复杂项目并行开发
- ✅ 24 小时不间断任务
- ✅ 需要本地文件/工具访问
- ✅ 隐私敏感场景

## 二、核心概念详解

### 2.1 任务规划 (Task Planning)

**三种模式**：

| 模式 | 描述 | 适用 |
|------|------|------|
| **静态编排** | 预定义执行顺序 | 流程固定、可预测任务 |
| **动态规划** | LLM 实时决策下一步 | 开放域、需灵活应对 |
| **混合模式** | 关键节点预设 + 中间自主 | 平衡控制与灵活性 |

**OpenClaw 实践**：
```markdown
1. 主 Agent 分析整体目标
2. 拆解为独立子任务（可并行）
3. 为每个子任务 spawn subagent
4. subagent 自主执行（不汇报中间状态）
5. 完成后 push 结果回主 Agent
6. 主 Agent 汇总/验证/交付
```

---

### 2.2 工具调用 (Tool Calling)

**调用机制**：
```
LLM 输出 → 解析工具名 + 参数 → 执行工具 → 返回结果 → LLM 继续推理
```

**OpenClaw 工具集**：
- `read/write/edit` - 文件操作
- `exec/process` - 终端命令
- `browser` - 网页自动化
- `feishu_*` - 飞书集成
- `web_search/web_fetch` - 网络搜索
- `message` - 消息发送
- `subagents` - 子 Agent 管理

**最佳实践**：
1. 工具命名清晰（动词 + 名词）
2. 参数校验在工具层完成
3. 错误信息友好可操作
4. 避免工具调用循环（设置最大次数）

---

### 2.3 自主决策 (Autonomous Decision Making)

**决策层级**：
```
L1: 单步决策（是否调用工具/调用哪个）
L2: 多步规划（任务拆解顺序）
L3: 策略选择（用什么方法完成任务）
L4: 目标调整（发现原目标不可行时修正）
```

**OpenClaw Subagent 自主权**：
- ✅ 自主选择工具调用顺序
- ✅ 自主决定重试/跳过策略
- ✅ 自主优化执行路径
- ❌ 不自主修改任务目标
- ❌ 不自主 spawn 下级 subagent

**安全边界**：
```markdown
- 外部操作（邮件/推文/公开）需确认
- 破坏性操作（删除/覆盖）需确认
- 隐私数据不外泄
- 不确定时询问主 Agent
```

## 五、实战建议

### 5.1 Subagent 使用原则

1. **任务独立性**：子任务应能独立执行，不依赖其他 subagent 中间状态
2. **结果 Push**：subagent 完成后主动汇报，主 agent 不轮询
3. **超时控制**：设置合理超时，避免卡死
4. **错误隔离**：单个 subagent 失败不影响整体

### 5.2 提效关键点

```markdown
✅ 用 subagent 并行化串行任务（节省 70% 时间）
✅ 用自动化工具替代手动操作（减少人为错误）
✅ 用模板化工作流减少重复决策（提升一致性）
✅ 用本地执行保护敏感数据（安全合规）
```

### 5.3 避坑指南

```markdown
❌ 避免 subagent 嵌套过深（最多 2 层）
❌ 避免 subagent 间直接通信（通过主 agent 协调）
❌ 避免无限循环（设置最大迭代次数）
❌ 避免过度并行（考虑资源限制）
```
